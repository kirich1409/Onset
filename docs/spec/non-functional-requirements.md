# Onset — Non-Functional Requirements (общие, обязательны для всех фич)

Кросс-каттинг требования. **Обязательны для каждой фичи** наравне с её AC. Любой feature-spec/PR проверяется против релевантных NFR; нарушение — блокер. Связано с [`overview.md`](overview.md) и [`architecture.md`](architecture.md).

Приоритет требований: **NFR-PERF, NFR-TEST, NFR-STACK, NFR-EXT — критичные** (явно названы как основа продукта). Остальные — обязательные.

---

## NFR-PERF — Производительность

Продукт — реалтайм-рекордер; производительность это контракт, не «по возможности».

- **PERF-1 (no-drops):** на референс-железе (M3 Max + 4K60) dual-stream 4K60+4K30 HEVC идёт без дропов в steady state. → AC-14, TC-30.
- **PERF-2 (hot path):** real-time callback не блокируется; только retain/enqueue; никаких actor-хопов на пути сэмплов; время удержания буфера < `minimumFrameInterval×(queueDepth−1)`. → architecture § hot path, TC-30 (signpost).
- **PERF-3 (память):** очереди capture→writer строго ограничены; peak footprint в пределах бюджета на 10-мин записи, без линейного роста. → TC-39.
- **PERF-4 (HW-энкод):** кодирование всегда аппаратное по умолчанию; software по умолчанию запрещён. → AC-16.
- **PERF-5 (латентность UX, калибруется на acceptance):** запуск → интерактивное окно настроек ≤ ~500 мс; нажатие Record → активная запись ≤ ~300 мс (warm-up учтён); превью камеры без видимых лагов.
- **PERF-6 (термал/энергия):** длинные записи мониторят `thermalState`; деградация вместо неконтролируемого throttle. → AC-15, TC-37.
- **Верификация:** hardware-acceptance L5 (TC-30/37/39/40) + os_signpost-замеры; perf-регрессии ловятся повторным прогоном acceptance-набора на референс-железе перед релизом.

## NFR-TEST — Тестируемость и автоматизированное тестирование

Сценарий разработки — автоматизированное тестирование на всех уровнях, где это возможно без железа, и документированный hardware-acceptance там, где железо обязательно.

- **TEST-1 (пирамида):** L1 статический (build/lint/typecheck) → L2 unit → L3 UI-instrumentation → L4 e2e/ui-scenario → L5 hardware-acceptance. Каждая фича декларирует, какие уровни применяет.
- **TEST-2 (автоматизация):** L1–L3 (и L4 где автоматизируемо) гоняются в CI на каждый PR (см. NFR-CI). L5 — документированный чек-лист на референс-железе.
- **TEST-3 (design-for-test):** чистые функции и протокол-швы как первичный механизм. Обязательно unit-тестируемы без железа: `Validator`, `SampleRouter` (fan-out + `isAlive`), atomic start (drop PTS<T, warm-up→T), gap-fill до fan-out, `CMSyncConvertTime`, **`DegradationLadder` как чистый decider-автомат**. Фейки: `EncodingWriter`/`CaptureSource`/`ClockProviding`. DI через composition root; никаких скрытых синглтонов на hot path.
- **TEST-4 (coverage-gate):** каждый публичный символ покрыт тестом или помечён trivial-no-test (data-class/enum). Критические пути (синхронизация, дропы, атомарный старт, валидация, маршрутизация) — 100% unit-покрытие. Метрика покрытия — не самоцель; гейт по публичному API и критическим путям.
- **TEST-5 (инфраструктура):** XCTest для unit/integration/UI-instrumentation; swift-snapshot-testing (additive) для визуальной регрессии UI; `manual-tester`/MCP-раннер для ui-scenario; скрипты пост-анализа файлов (`ffprobe`/`AVAsset`, см. [`testing.md`](testing.md) Appendix A) для L5.
- **TEST-6 (регрессии):** автор изменения, ломающего тесты, чинит их в том же PR; `@Ignore` без ссылки на трекер запрещён.
- **Верификация:** наличие тестов на каждый AC (см. per-feature test-plans), зелёный CI, coverage-gate в пайплайне.

## NFR-STACK — Максимально нативный стек

- **STACK-1:** только нативные Apple-фреймворки (ScreenCaptureKit, AVFoundation, AVFAudio, AVAssetWriter, VideoToolbox, Core Media, AppKit/SwiftUI). Сторонние зависимости **запрещены без явного plan-stage согласования** (исключения для тест-инфры — swift-snapshot-testing — допустимы как dev-only).
- **STACK-2:** Swift, deployment target macOS 26.0, Apple Silicon; использовать актуальный SDK, верифицировать API против него (architecture § верификация SDK).
- **STACK-3:** никаких враппер-слоёв поверх системных API «на всякий случай» — Domain говорит на языке CoreMedia на hot path (сознательная граница).
- **Верификация:** аудит зависимостей в CI (нет сторонних runtime-deps); review плана при попытке добавить зависимость.

## NFR-EXT — Расширяемость и масштабируемость архитектуры

Архитектура закладывается под будущие фичи и форматы СРАЗУ. Принцип **open/closed**: новая фича/формат добавляется через новую реализацию + регистрацию, без правки ядра.

- **EXT-1 (стабильные швы):** Domain-протоколы (`CaptureSource`, `EncodingWriter`, `SampleSink`, `ClockProviding`) — точки расширения. Новые источники/выходы реализуют протокол, не меняя координатор.
- **EXT-2 (реестры расширения):**
  - **Источники захвата** — новый `CaptureSource` (будущее: дополнительные камеры, системный звук, виртуальные источники).
  - **Кодеки/форматы вывода** — codec-registry (сейчас HEVC/H.264; будущее: ProRes archival, AV1-когда-encode, новые контейнеры) через `EncodingWriter` + конфиг, без правки пайплайна.
  - **Режимы захвата** — `CaptureScope` enum (сейчас `.fullDisplay`; Phase 2: `.region`, `.window`) — добавление варианта не ломает downstream.
  - **Шаги деградации** — `DegradationLadder` расширяется новыми шагами.
  - **UI-поверхности** — новые экраны добавляются как отдельные SwiftUI-вью поверх ViewModel.
- **EXT-3 (модульность):** структура `docs/<feature>/` зеркалит модули кода — каждая фича вертикальный срез; направление зависимостей (внутрь) контролируется. Будущая фича = новая папка + новый модуль + объявленные связи.
- **EXT-4 (forward-compat форматов):** в выходные файлы/метаданные закладывается версионирование (например, app/format version в metadata), чтобы новые форматы и поля сосуществовали со старыми без поломки потребителей.
- **EXT-5 (без преждевременной абстракции):** расширяемость через существующие протоколы и enum'ы, а не спекулятивные плагин-фреймворки. Точки расширения — те, что названы выше; новые добавляются по мере реальной потребности.
- **Верификация:** архитектурный review при добавлении фичи/формата — проверка, что ядро не правится; направление зависимостей не нарушено.

## NFR-ERR — Обработка ошибок и crash-safety

- **ERR-1 (no silent failures):** ни одна ошибка не глотается; дропы вскрываются (AC-21), отказы уведомляются (AC-17/20). Единая политика ошибок.
- **ERR-2 (crash-safety файлов):** `AVAssetWriter` с `movieFragmentInterval`, чтобы файлы оставались воспроизводимыми после краша/force-quit (могут быть обрезаны, не corrupted). → TC-41.
- **ERR-3 (isolate):** отказ одного writer'а/источника не рушит остальные (`isolateAndContinue`). → AC-17/20.
- **ERR-4 (graceful edge):** диск переполнен, отзыв permission, unplug устройства, sleep/wake — корректная обработка с уведомлением, без потери данных. → TC-35/26/42.
- **Верификация:** TC-24/25/26a/26b/35/41/42; review error-handling (silent-failure-hunter при необходимости).

## NFR-CI — CI/CD и quality gates

- **CI-1 (merge-гейт на PR):** автоматически build + lint (swift-format/SwiftLint) + unit (L2) + UI-instrumentation (L3) — всё зелёное обязательно для merge.
- **CI-2 (coverage-гейт):** проверка покрытия публичного API / критических путей (NFR-TEST).
- **CI-3 (hardware-acceptance):** L5 не гоняется в стандартном CI (нужно физическое железо/дисплей/камера) — отдельный документированный чек-лист/лаб-прогон перед релизом.
- **CI-4 (release):** code signing + notarization для распространения; семантическое версионирование; changelog.
- **CI-5 (deps-аудит):** проверка отсутствия несанкционированных сторонних runtime-зависимостей (NFR-STACK).
- **Верификация:** конфигурация CI в репозитории; обязательные статусы на PR.

## NFR-SEC — Безопасность и приватность

Запись экрана = потенциально чувствительные данные (пароли, личная переписка на экране).

- **SEC-1 (local-only):** никакой сети, телеметрии, аналитики по умолчанию. Записи никогда не покидают устройство.
- **SEC-2 (least privilege TCC):** запрашиваются только нужные разрешения (screen/camera/mic/notifications); понятные denied-состояния. → AC-18.
- **SEC-3 (безопасная работа с файлами):** запись в выбранный пользователем путь; валидация пути; никакого кэша кадров на диск вне целевого файла; не индексировать/не шарить записи.
- **SEC-4 (supply-chain):** отсутствие сторонних runtime-SDK снижает риск (следствие NFR-STACK).
- **SEC-5 (sandbox-ready):** если в будущем включается App Sandbox — security-scoped bookmarks для выбранной папки; entitlements камеры/микрофона.
- **Верификация:** review security-expert при работе с файлами/правами; проверка отсутствия сетевых вызовов.

## NFR-A11Y — Доступность

- **A11Y-1:** окно настроек и menu bar полностью управляются с клавиатуры (tab order, focus ring, key-equivalents); остановка записи доступна с клавиатуры (hotkey). → AC-19, TC-38.
- **A11Y-2:** VoiceOver читает все контролы, disabled-состояния во время записи, индикатор записи и счётчик дропов; признак деградации передаётся **не только цветом**. → TC-38.
- **A11Y-3:** Dynamic Type где применимо; достаточный контраст (WCAG AA).
- **Верификация:** TC-38 (ui-instrumentation/ui-scenario с VoiceOver); ux-expert review.

## NFR-I18N — Локализация (i18n-ready с самого начала)

- **I18N-1:** все user-facing строки вынесены в String Catalog (`.xcstrings`)/`.strings`; никакого хардкода UI-текста в коде.
- **I18N-2:** locale-корректное форматирование дат/времени (имена session-папок, таймер), чисел.
- **I18N-3:** layout устойчив к разной длине строк (без обрезания/перекрытия).
- **Примечание:** фактические переводы вне MVP; требование — **готовность** к локализации (извлекаемость строк), чтобы не переписывать UI позже.
- **Верификация:** lint/review на отсутствие хардкод-строк; pseudo-localization прогон (опц.).

## Применение

- Каждый feature-spec наследует эти NFR; в его test-plan релевантные NFR отражаются конкретными TC (perf → L5 TCs, a11y → TC-38, error → TC-24/41 и т.д.).
- `/finalize` и `/acceptance` проверяют код против релевантных NFR; `multiexpert-review` профиля spec/test-plan — против NFR при наличии триггеров (perf-critical, a11y, security).
