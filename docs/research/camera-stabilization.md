## TL;DR

| Вопрос | Ответ |
|---|---|
| Есть ли готовый API-путь («включить флаг») | **Нет.** AVFoundation-стабилизация — iOS/Catalyst-only |
| Нужны ли сторонние зависимости | **Нет.** Всё на Apple-фреймворках (Vision/CoreImage/VideoToolbox) |
| Оценка движения | `VNTranslationalImageRegistrationRequest` на **3× апскейле** пары кадров (0.33 px) |
| Алгоритм сглаживания | каузальный lock-with-slow-recenter (rate-limited reference), без look-ahead |

| Кроп-бюджет | ~0.5% на сторону (8/16 px @1080p), не «индустриальные» 5% |
| Перф @1080p (M3 Max) | 31.2 мс/кадр p50 (оценка+рендер @3×) — влезает в каденцию Brio (20–25 fps → 40–50 мс) |

## Проблема

Веб-камера (Logitech MX Brio, UVC, гироданных нет) стоит на мониторе; вибрация от набора текста
на клавиатуре передаётся через стол/монитор в камеру — в записи видна мелкая высокочастотная
тряска (1–3 px за кадр). Задача — программная стабилизация camera-потока в пайплайне записи
Onset, опциональная, без стороннего железа.

## Почему нет готового пути

**AVFoundation-стабилизация на native macOS отсутствует.** Проверены все 5 релевантных символов
через apple-docs MCP — в каждом platform line содержит iOS/iPadOS/tvOS/Mac Catalyst, но не bare
macOS (консистентный паттерн `API_UNAVAILABLE(macos)`):

- `AVCaptureConnection.preferredVideoStabilizationMode`
- `AVCaptureConnection.activeVideoStabilizationMode`
- `AVCaptureVideoStabilizationMode`
- `AVCaptureDevice.Format.isVideoStabilizationModeSupported`
- `AVCaptureDevice.isVideoStabilizationSupported` (устаревший)

**В `VTFrameProcessor` (macOS 15.4+) stabilization-процессора тоже нет** — полный каталог
эффектов ограничен frame-rate conversion, low-light interpolation, super resolution, motion
blur, optical flow, temporal noise filter.

**Готовых поддерживаемых сторонних решений в нише нет.** Единственный open-source real-time
аналог для macOS (LiveVisionKit, OBS-плагин) архивирован с 2024-04. Camo (Reincubate) имеет
фичу «Stabilise video» ровно под typing-wobble сценарий и подтверждает, что задача решаема в
real-time — но это сторонний платный софт, не встраиваемое решение.

Вывод: стабилизация — собственный этап пайплайна на строительных блоках Apple (Vision /
CoreImage / VideoToolbox), сторонние зависимости не нужны и не оправданы.

## Наш кейс — «лёгкий» случай EIS

Камера статична, вибрация имеет нулевое среднее вокруг неподвижной позиции, намеренного
движения (панорамы) нет. Это означает: сглаживать траекторию камеры не нужно — достаточно
**lock-to-reference**: привязка к медленно обновляемому опорному кадру + малый запас кропа.
Это одновременно и лучший сценарий для EIS (low-intensity high-frequency fixed vibration —
industry consensus, IPVM), и алгоритмически самый простой (каузальный фильтр без look-ahead).

## Кандидаты оценки движения

Четыре кандидата сравнены эмпирически на реальной записи (M3 Max, 1080p60 CFR, тряска от
набора текста):

| Кандидат | Субпиксельность | Латентность @1080p | Вердикт |
|---|---|---|---|
| `VNTranslationalImageRegistrationRequest` (native, без апскейла) | **целочисленная** (0 дробных из 9537 пар) | 10.4 мс p50 | недостаточно точен при тряске 1–3 px |
| `VNTranslationalImageRegistrationRequest` @2× апскейл | 0.5 px | 14.8–15.5 мс p50 | хорош, квантование 0.5 px оставляет lock-to-ref хвост до ~2 px |
| `VNTranslationalImageRegistrationRequest` @3× апскейл | **0.33 px** | 26.4–29.6 мс p50 | **выбран** — квантовый шум ниже измерительного пола |
| `VNTranslationalImageRegistrationRequest` @4× апскейл | 0.25 px | 37.3 мс p50 | точнее не нужно, латентность лишняя |
| `VNHomographicImageRegistrationRequest` | континуальный (float warp-матрица) | — | инжектит паразитное движение до 4.8 px на статичной сцене, 10.8% выбросов на движении → только как fallback с гейтингом |
| `VTMotionEstimationSession` (macOS 26.0, новый API) | 0.5 px, но только в конфиге `size=4 + multipass` | 28.5–28.7 мс p50 @1080p | **отклонён** — см. ниже |

### Почему отклонён `VTMotionEstimationSession`

Новый API ровно macOS 26.0, doc-страницы с пустыми Overview — форма и семантика проверены
эмпирически (`vtformat.swift`, `vtmotion.swift`):

- Выходной буфер — `CVPixelBuffer` формата `2C0h` (`kCVPixelFormatType_TwoComponent16Half`):
  два `Float16`-лейна (dx, dy) на block. Субпиксельность подтверждена, но **только** в
  конфигурации `MotionVectorSize=4` + multipass — грубые конфиги (`size=16`, `multipass=false`,
  1.7–2.1 мс p50) дают только целочисленную точность на блок 16×16 px.
- **На 4K конфиг `size=4 + multipass=true` не создаётся** — `__VTMotionEstimationSessionCreate`
  возвращает `kVTParameterErr` (−12902). На 4K доступен только грубый `size=16` (5.86 мс p50,
  целочисленный).
- Латентность точного конфига (28.5–28.7 мс p50 @1080p, до 34.7 мс max) не даёт запаса против
  `VNTranslationalImageRegistrationRequest@3×` (26.4–29.6 мс) при худшей субпиксельности и без
  доступности на 4K.
- Confidence-канала на block нет — нельзя отличить надёжный вектор от шумного без
  дополнительной эвристики.

`VNTranslationalImageRegistrationRequest@3×` (2× для 60 fps-источников) остаётся основным
выбором: масштаб апскейла — свободный параметр этапа, подбираемый под каденцию источника.

## Выбранный подход

Real-time этап в пайплайне записи, camera-only, аддитивный к текущей архитектуре:

- **Декоратор `StabilizingVideoSource` над `VideoFrameSource`**, между `CameraSource` и
  `VideoEncoder` (шва «между CFR и энкодером» не существует — `CFRNormalizer` живёт внутри
  `VideoEncoder`; до энкодера этап видит только реальные кадры, hold-repeat CFR синтезируются
  позже и не отравляют оценку движения).
- Пара по паттерну проекта: **pure nonisolated `StabilizationSmoother`** (образец —
  `CFRNormalizer`, тестируем на L2) + **impure actor с GPU-контекстом** (образец —
  `LiveCompressionSession`).
- **Smoother: lock-with-slow-recenter (rate-limited reference).**
  `ref += clamp(alpha·(cum − ref), ±maxRefStep)`; `correction = ref − cum`.
  Параметры прототипа: `alpha=0.05`, `maxRefStep=0.01 px/кадр` (= 0.6 px/s @60fps),
  `deadband=0`. Плоская EMA (любой `alpha` 0.005–0.02) упирается в лаг (lock-to-ref max
  1.8–2.3 px) — rate-limit решает: всплеск не может тащить опору (шаг клампится), медленный
  дрейф реальной записи (~0.06 px/s) проходит с 10× запасом. Deadband оставлен только как
  телеметрия «доля кадров с нулевой коррекцией» — сам по себе не помогает, рендер идёт каждый
  кадр (см. красные флаги).
  Знак коррекции: **`correction = −alignmentTransform`** — валидирован эмпирически (со знаком
  «+» residual растёт 2×, с «−» падает 4–20×; должен быть закреплён тестом в реализации).
- **Геометрия: session-fixed crop.** `(16, 9, 1888, 1062)` — ровно 16:9, чётные размеры;
  изотропный scale-back в `1920×1080` (×1.016949). Margin 16/9 px покрывает измеренный max
  отклонения 4.84 px с 2–3× запасом. Адаптивный кроп на кадр невозможен — `VideoToolbox`
  требует неизменных dimensions на сессию; фиксируется в `ResolvedCameraPlan`.
- **Рендер каждого кадра в НОВЫЙ `CVPixelBuffer` из собственного `CVPixelBufferPool`** —
  инвариант read-only `VideoFrame` (`PipelineTypes.swift:113`) сохраняется, входной буфер не
  мутируется, PTS исходные (без пере-конвертации host-clock).
- **Каузальность обязательна** — без look-ahead: для lock-to-reference её достаточно; окно
  look-ahead 15 кадров @4K дало бы ≈190 МБ удержанных буферов и риск истощения capture-пула.
- **Телеметрия и деградация**: новый case в `DropSource`, множитель в `EngineBudgetCap` /
  `CapabilityResolver`, рычаг деградации «выключить стабилизацию» раньше ужимания разрешения.
  Screen-пайплайн этап не затрагивает.

## Результат прототипа (spike #295)

Материал: реальная запись 1080p60 CFR, 81.6 s, 4769 кадров, тряска от набора текста
(MX Brio). Полный этап (оценка @3× → каузальный smoother → CI-рендер) прогнан на всей записи.

### До / после

| Метрика | Baseline | Стабилизировано (@3×) |
|---|---|---|
| Движущихся пар (субпиксельный измеритель @2×) | 19.0% | **0.2%** |
| Per-pair сдвиг p50/p95/max, px | 0.5 / 1.5 / 3.5 | 0.5 / 0.5 / 0.5 (пол измерителя) |
| Lock-to-ref (2 s rolling) p95/p99/max, px | 2.00 / 3.96 / 4.84 | **0.36 / 0.50 / 0.81** |
| Движущихся пар (целочисленный measure-shake) | 9.7% (461 пара) | **0.00% (0 пар)** |

**Критерий эпика «lock-to-ref residual ≤1 px» выполнен измеренно** (max 0.81 px), не только
теоретически. Теоретический residual самого этапа (из телеметрии ref-траектории): max 0.60 px.

### Перф @1080p (M3 Max, полный прогон, warmup исключён)

| Компонент | @2× | @3× |
|---|---|---|
| Оценка (апскейл + Vision), p50/p95 | 15.5 / 18.6 мс | 29.6 / 33.1 мс |
| Рендер (CI translate+crop+scale, BGRA), p50/p95 | 1.4 / 2.1 мс | 1.5 / 2.0 мс |
| Итого, p50 | 17.0 мс | 31.2 мс |
| Peak RSS процесса | 82 МБ | 122 МБ |

Теория (не мерено на реальном 4K-видео): на 4K-записи оценка на рабочем 1080p-даунскейле даёт
ту же стоимость оценки, рендер дороже ~4× по пикселям (ожидаемо ~6 мс) — укладывается в бюджет.

### Визуальный вердикт

Side-by-side клипы 3 самых тряских сегментов и покадровые PNG-гриды (соседние кадры, ×3
nearest) осмотрены вручную: кромки чисты, двоения/смаза ресемплинга нет, геометрия стабильна.
Лёгкий zoom 1.7% от кропа — ожидаемый и незаметный без прямого сравнения с оригиналом.

## Красные флаги для реализации (#296/#297)

1. **«Skip render» несовместим с session-fixed crop.** Passthrough сырого буфера при нулевой
   коррекции даёт zoom-flicker (кроп/масштаб меняются между кадрами) — рендерится каждый кадр
   без исключений. Стоимость этапа на статичной сцене = render-cost ≈ 1.4–1.5 мс/кадр + оценка.
2. **Прототип рендерит в BGRA** — production обязан рендерить в NV12/420v (нативный формат
   `VTCompressionSession`), иначе лишняя конверсия форматов в энкодере.
3. **Масштаб оценки — параметр этапа, не константа.** @3× (29.6 мс) не влезает в бюджет кадра
   60 fps (16.7 мс) — для 60 fps-источников использовать @2× либо принять запаздывание
   коррекции (пайплайн асинхронный, `.bufferingNewest(4)` поглощает).
4. **Латентность Vision имеет редкие выбросы** (85–250 мс на первых кадрах / переключении
   планировщика) — этап обязан быть асинхронным с дроп-политикой, не блокировать кадровый
   поток (текущая архитектура это уже поддерживает).
5. **Знак коррекции** — `correction = −alignmentTransform`; закрепить unit-тестом в реализации,
   не полагаться на повторную эмпирическую проверку.
6. **Sandbox-ловушка для CLI-инструментов приёмки**: `AVAssetReader` из sandboxed-процесса
   падает с `-11800`/`-17913` — инструменты замера (`measure-shake`, verify-скрипты) нужно
   запускать unsandboxed.

## Ограничения

Стабилизация **не чинит**:

- **Motion blur внутри кадра** — у Brio уже известный low-light смаз (см.
  `docs/quality/production-quality-bar.md`), отдельная проблема экспозиции камеры, не геометрии.
- **Rolling-shutter jello** — кроп потенциально делает его заметнее, а не менее заметным.

Эффект стабилизации нагляден именно на резких кадрах при нормальном освещении и вибрации от
клавиатуры/стола — ровно целевой сценарий эпика #294. В темноте выигрыш от стабилизации
съедается уже существующим motion blur.

## Открытые вопросы (не блокируют spec, но требуют решения в дизайне)

- Точная стоимость GPU-рендера сдвига на реальном 4K-материале (прототип — только теория ×4 по
  пикселям, не измерено).
- Поведение этапа под живым backpressure пайплайна записи (прототип — оффлайн CLI над готовым
  файлом, не встроен в `RecordingSession`).
- Энергия/теплопакет при длинной записи — первая постоянная GPU-нагрузка в приложении,
  отдельная оценка на этапе дизайна (см. #296).
